인사 자원 시스템 — 멀티 테넌시 아키텍처 설계

2024.07 ~ 2024.12 · 주식회사 디케이테크인

배경

다수의 조직(테넌트)이 하나의 인사 자원 관리 시스템을 공유해야 하는 요구사항이 있었습니다. 각 조직마다 별도 시스템을 구축하면 유지보수 비용이 기하급수적으로 증가하고, 조직 추가 시마다 인프라를 별도로 구성해야 하는 구조적 한계가 있었습니다. 제한된 비용과 일정 안에서 데이터 격리, 인증 분리, 주기적 데이터 동기화까지 하나의 아키텍처로 해결해야 하는 과제였습니다.

문제 정의

  • 조직마다 독립적인 데이터 격리가 필요하지만, 인프라 비용과 개발 일정을 최소화해야 함
  • 새로운 조직 추가 시 코드 변경 없이 확장 가능한 구조가 필요
  • 멀티 테넌시 구조에서 테넌트별 주기적 데이터 동기화(공휴일, 급여 마감일 등)를 서비스에 영향 없이 처리해야 함

해결 과정

1. 테넌트 분리 전략 — Shared Database + Shared Schema 선택

멀티 테넌시 구현 방식은 크게 세 가지가 있었습니다. DB 인스턴스 분리는 격리성이 높지만 테넌트 수만큼 DB를 운영해야 해서 비용과 관리 부담이 컸고, 스키마 분리는 JPA 설정 복잡도가 높아 제한된 일정 안에 안정적으로 구현하기 어렵다고 판단했습니다. 비용과 시간을 고려했을 때 컬럼 분리 방식이 가장 현실적이었고, 단일 DB에서 TENANT_ID / COMPANY_ID 컬럼으로 데이터를 격리하는 Shared Database + Shared Schema 방식을 선택했습니다.

애플리케이션 레벨의 쿼리 필터링(WHERE COMPANY_ID)으로 데이터를 격리하고, 단일 HikariCP 커넥션 풀로 운영하여 리소스 관리를 단순화했습니다. 다만 이 방식은 WHERE 조건 누락 시 다른 테넌트의 데이터가 노출될 수 있는 위험이 있어, 코드 리뷰 시 쿼리 필터링 누락 여부를 중점적으로 검토하는 프로세스를 마련하여 보완했습니다.

테넌트 분리 방식 비교 → Shared DB + Shared Schema 채택

DB 분리
비용 ↑ 관리 부담 ↑ vs 스키마 분리
JPA 복잡도 ↑ 일정 초과 vs 컬럼 분리 ✓
비용 ↓ 확장성 ↑

2. 인증 및 세션 — 테넌트별 격리 설계

Keycloak OIDC를 통한 인증 후 OidcUserDto에서 companyId를 추출하여 UserDetail에 주입하는 구조로, 인증 시점부터 테넌트 식별이 자연스럽게 이루어지도록 했습니다. 세션은 Redis에 테넌트별 네임스페이스(dkt-ehr:session:{tenant}:user)를 적용하여 테넌트 간 세션 충돌을 방지했습니다.

인증 → 테넌트 식별 흐름

Keycloak OIDC → OidcUserDto → UserDetail (companyId) → 쿼리 필터링

3. 테넌트별 배치 처리 — Spring Batch + @Scheduled

멀티 테넌시 구조에서 테넌트별로 주기적인 데이터 동기화(공휴일, 급여 마감일, 증명서 만료 등)가 필요했고, 이를 별도 schedule-module로 분리하여 Spring Batch 기반으로 처리했습니다. 서비스 모듈에 영향을 주지 않으면서 테넌트별 데이터를 순차적으로 처리하는 구조입니다.

결과

  • 단일 DB 구조로 인프라 비용을 최소화하면서 다수 조직의 데이터 격리 확보
  • Keycloak 인증 + Redis 세션 네임스페이스로 인증 및 세션 레벨의 테넌트 격리 보장
  • 배치 모듈 분리로 주기적 동기화 작업이 서비스 안정성에 영향을 주지 않는 구조 확보
  • 신규 테넌트 추가 시 설정 기반 테넌트 식별로 빠른 온보딩 가능

회고

멀티 테넌시 설계에서 가장 중요했던 것은 "완벽한 격리"가 아닌 "상황에 맞는 격리"를 선택하는 것이었습니다. DB 분리 방식이 격리성 면에서는 우수하지만, 프로젝트의 비용과 일정을 고려하면 Shared Schema + 애플리케이션 레벨 필터링이 더 현실적인 선택이었습니다. 물론 이 방식은 쿼리 필터링 누락이라는 잠재적 위험이 있기 때문에, 코드 리뷰를 통한 지속적인 검증이 필수적이라는 점도 함께 배울 수 있었습니다.
